# GridSight UK — Azure deployment

Data stays on **HuggingFace** (the bronze parquet lake). Azure hosts only **compute +
serving**. The forecast is refreshed **hourly** and served by a small always-on API; a
static dashboard reads that API.

```
 HuggingFace Hub                         Azure
┌──────────────┐  sync  ┌───────────────────────────────────────────┐
│ bronze/*.pqt │◄─────► ┤ Container Apps JOB  (cron: 0 * * * *)       │
│ (data lake)  │  push  │  gridsight-serve — model pulled from HF      │
└──────────────┘        │  fresh tail → push bronze → silver → forecast│
 live sources ─────────►│        │ writes forecast_*.json             │
  Met AWS S3  ─────────►│   ┌────▼──────┐                             │
  PV_Live API           │   │Azure Files│  share "serve"              │
  NESO CKAN             │   └────┬──────┘                             │
┌──────────────┐        │        │ reads                              │
│  Key Vault   │─token─►│  ┌─────▼──────────┐                         │
│  (hf-token)  │  (MI)  │  │ Container App  │  FastAPI, always-on     │
└──────────────┘        │  │ gridsight-api  │─── /forecast ───►       │
                        │  └────────────────┘        Static Web App   │
                        └───────────────────────────────────────────┘
```

## Why this shape

| Decision | Choice | Reason |
|---|---|---|
| Data lake | HuggingFace (unchanged) | Already implemented, free, `snapshot_download` is incremental. Swap to ADLS via fsspec later for "prod". |
| Fresh data | Job pulls the **tail** from sources, then **pushes new bronze back to HF** | Met NWP (3h), PV_Live (30m), NESO (1h). Bronze must persist — Met AWS S3 only keeps recent inits. Silver/Gold are rebuilt from bronze each run; the **weekly retrain** also syncs them to HF (hourly would re-upload the full rebuild). |
| Handoff | Azure Files share | Job writes / API reads the 4 `forecast_*.json`. Tiny, standard Container Apps pattern. |
| Serve cadence | Hourly (`0 * * * *`) | Matches NESO+PV_Live, catches every NWP run regardless of publish lag. |
| Secrets | Key Vault + Managed Identity | HF token never in an image or plaintext env. |
| Registry | ACR + `az acr build` | Images build in the cloud — **no local Docker required**. |
| Frontend | Vanilla `index.html` → Static Web Apps (Free) | One page reading one API; a framework would be overkill. |

## What runs where — and when

Three separate things; only two are containers, and only one does heavy work.

| Component | Docker? | Runs when | Weight |
|---|---|---|---|
| **Dashboard** (`frontend/`) | No | Static site on Static Web Apps — always available | Just HTML + `history.json` |
| **API** (`gridsight-api`) | Yes | **Continuously, 24/7** (always-on Container App) | **Light** — only reads the forecast JSON; no model/torch |
| **Serve job** (`gridsight-job`) | Yes | **Only on the hourly cron** — spins up → runs → **exits** | **Heavy** — fetch fresh data, silver→gold, run the model, write JSON |

So the heavy compute (torch / LightGBM / the data pipeline) is **not** running all the
time — the job container starts each hour, works for a few minutes, then shuts down
(billed per execution-second). The API container stays up so requests are instant, but
it is tiny because it never touches the model — it just serves the JSON the job produced.

```
        every hour (cron)                       always up
   ┌────────────────────┐   writes JSON   ┌────────────────────┐
   │ JOB container      │──►  Azure  ────►│ API container      │──► Dashboard
   │ (start → run → exit)│    Files        │ (reads JSON, light)│    (static, no Docker)
   └────────────────────┘                 └────────────────────┘
     torch / lightgbm here                  FastAPI file-reader
```

Two things people expect to be Docker but aren't:

- **`history.json`** (the dashboard's 2-year backtest) is generated by running
  `frontend/build_history.py` **directly with Python**, not in a container. That is a
  one-off demo asset — separate from the job, which produces the *next-12h live* forecast.
- **You don't need Docker on your machine.** `az acr build` builds the images in the
  cloud. Locally you run everything with plain Python (`env/bin/python …`); Docker is
  only the runtime packaging for Azure.

## Prerequisites

- Azure CLI (`az login` done) with a subscription that can create resource groups.
- A HuggingFace token with **write** access to your bronze repo (the job appends fresh
  bronze back to HF each run; set `GRIDSIGHT_PUSH_BRONZE=0` if you only have read).
- (Frontend only) Node.js, for the SWA CLI via `npx`.

Local Docker is **optional** — only needed for `docker-compose` smoke tests.

## Deploy

```bash
# 1. configure
cp deploy/scripts/config.env.example deploy/scripts/config.env
$EDITOR deploy/scripts/config.env          # RG, LOCATION, PREFIX, ACR, HF repo
export GRIDSIGHT_HF_TOKEN=hf_xxxxxxxx       # kept out of the file / git

# 2. provision everything (RG, ACR, images, Key Vault, storage, env, API, jobs)
./deploy/scripts/deploy.sh

# 3. seed the first forecast now (don't wait for the top of the hour)
./deploy/scripts/run_serve_now.sh

# 4. publish the dashboard to Static Web Apps (pass the API URL for the live overlay)
env/bin/python frontend/build_history.py            # regenerate history.json if the model changed
./deploy/scripts/deploy_frontend.sh https://<api-fqdn>
```

The dashboard is a **history explorer**: it loads `frontend/history.json` (2 years of
out-of-sample forecast vs actual vs NESO, stack + Chronos) and lets you scrub/zoom by
window. Passing the API URL to `deploy_frontend.sh` also **overlays the real next-12h
forecast** at the right edge (a dashed "now" line + "LIVE" region), fetched live from
`/forecast`. Omit it for a history-only, offline-friendly page.

`deploy.sh` is idempotent — re-run it after changing schedules or params. After a **code**
change, `./deploy/scripts/build.sh` rebuilds the images and rolls them out.

## First-run backfill (important)

The hourly job only tops up a short window of fresh data (`GRIDSIGHT_BRONZE_LOOKBACK_DAYS`,
default 7). If your HF bronze is older than that, run a one-off wider backfill once so the
feature history is continuous. Either bump the env var on the job for a single run, or run
the bronze ingest locally and `upload_to_hf` before the first serve.

## Model registry (retrain → promote → serve)

Models live in an HF **model repo** (`GRIDSIGHT_MODEL_HF_REPO`, else derived from the
bronze repo: `…-bronze` → `…-model`), one folder per horizon:

```
<user>/gridsight-model
  model_12h/{stack.joblib, tcn.pt, metrics.json}
  model_6h/ {stack.joblib, tcn.pt, metrics.json}
```

- **Serve** pulls the live model from HF at load time (`modeling/registry.py:pull_model_dir`),
  falling back to the image's baked-in model if HF is unset/unreachable
  (`GRIDSIGHT_MODEL_FROM_HF=0` forces baked-only).
- **Retrain** trains, then **promotes only if better**: it uploads the new model **only
  when its test `mean_pinball` beats the current HF model's** (`push_model_if_better`).
  A bad retrain is kept out of production; HF git history retains every promoted version.

So retrain now actually takes effect (no image rebuild needed), and can't silently
regress the live model.

## Follow-ups (not in this deploy)

- **Incremental silver/gold** (rolling window) so the pipeline is cheap enough to run more
  often than hourly. Today it rebuilds ~2 years each run.
- **Incremental history**: `history.json` is a static 2-year backtest regenerated on
  demand; the live overlay only appends the next 12h. Auto-refreshing it (e.g. nightly)
  would keep the backtest tail current without a redeploy.
- Coverage calibration (76%→80%), rolling-origin CV.
